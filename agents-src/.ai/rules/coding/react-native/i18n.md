name: react-native-i18n
Scope: When adding user-facing strings, translating messages, or configuring internationalization
description: react-i18next with bundled JSON resources and device-locale detection via expo-localization; no HTTP backend
---

Use `react-i18next` for all internationalization — same library as the web stack. Never hardcode user-facing strings.

## Setup

```bash
npx expo install expo-localization
npm install i18next react-i18next
```

Unlike the web stack, translation files are bundled at build time, not fetched over HTTP — there is no static file server on a device, so `i18next-http-backend` is not used. Locale is detected from the device, not the browser:

```ts
// src/lib/i18n.ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import { getLocales } from 'expo-localization'
import ptBR from '../../assets/locale/pt-BR/translation.json'
import en from '../../assets/locale/en/translation.json'

const deviceLanguage = getLocales()[0]?.languageTag ?? 'pt-BR'

i18n.use(initReactI18next).init({
  resources: { 'pt-BR': { translation: ptBR }, en: { translation: en } },
  lng: deviceLanguage,
  fallbackLng: 'pt-BR',
  interpolation: { escapeValue: false },
})

export default i18n
```

Import once, before the root layout renders (see `Bootstrap`):

```ts
// app/_layout.tsx
import '@/lib/i18n'
```

## TypeScript key safety

Identical pattern to the web stack:

```ts
// src/lib/i18n.d.ts
import ptBR from '../../assets/locale/pt-BR/translation.json'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation'
    resources: { translation: typeof ptBR }
  }
}
```

## Locale files and key organization

```
assets/locale/
  pt-BR/
    translation.json
  en/
    translation.json
```

Organize keys by screen and component — never by domain/DDD concept — identical rule to the web stack. See `.ai/rules/coding/react/i18n.md` for the full key-shape example; it applies unchanged.

## Changing translations

Because resources are imported and bundled (not fetched), editing a translation `.json` file requires a Metro reload to take effect — it is not picked up live the way the web stack's `public/locale/` files are. A translation-only change still does not require a new native build; it ships fine via EAS Update (see `Build & Release`).
